What Is the Ingest System?

Keeping the KML support wiki in sync with source code — automatically

The KML support wiki (wiki-repo) documents what is enforced in lab accounts — config-rule limits, abuse thresholds, allowed services, IAM policies. This content is derived from source code in source-repo.

Without automation, the wiki drifts. A developer renames a function, changes a limit, or removes a config rule — and the wiki still shows the old information. Students get incorrect guidance. Support agents work from stale docs.

The ingest system solves this: every push to source-repo on Source VCS triggers a GitHub Actions workflow that detects which wiki pages are affected, fetches the latest source, recompiles those pages, and commits the result — automatically.

Two approaches have been explored

Graph-Based (Production)

Uses a semantic knowledge graph built from wiki content. Traverses graph edges to find related pages. Relies on LLM-inferred relationships between concepts.

LSP-Based (New)

Uses a Language Server Protocol to analyse the source code structurally. Traces actual code execution paths — which functions are called, which are dead, which symbols are referenced.

Authentication

GitHub Actions authenticates to Source VCS via a deploy token (KML_GITLAB_DEPLOY_TOKEN) stored as a GitHub repository secret. This token is used in two places:

1. Sparse clone (LSP workflow)

Authenticates the git clone of source-repo from Source VCS to build the local source tree for pylsp indexing.

2. Source VCS API fetch (both workflows)

The skill fetches individual source files (config rules, thresholds, policies) via curl to the Source VCS API using this token at compile time.

If this token expires or is rotated, both workflows will fail silently — the sparse clone will be refused and API fetches will return 401. The token must be kept current in GitHub Actions secrets.

The following pages walk through each approach, the test results, and what the evidence shows.

Why sparse clone?

pylsp needs source files on disk to index them. platforms/AWS/ contains all Python config rules, thresholds, and account logic. Policy JSON files and frontend code are excluded — LSP cannot do find_references on JSON, and they are not needed. The sparse checkout keeps CI fast and lightweight.

Current scope gap

Sparse checkout currently covers platforms/AWS/ only. Azure and GCP Python files are not on disk — LSP falls back to grep for Azure/GCP changes. Expanding to platforms/Azure platforms/GCP is the next step pending go-ahead.

What the Results Tell Us

The pattern that emerges from the three test cases

Across three test cases, one consistent difference emerged between the two approaches:

Graph answers

Which wiki pages are related to this changed file?

Semantic proximity — LLM-inferred

LSP answers

Which wiki pages document something that is actually executed?

Code-structural — compiler-verified

For a support wiki that documents what is enforced in lab accounts, the distinction matters. A function can exist in source without being called. The graph cannot tell the difference. LSP can — find_references returning zero callers is an unambiguous structural fact.

Graph — s3_limits.py

  • Sees check_s3_bucket_count defined
  • Infers: "S3 limits exist → update wiki"
  • Writes: max 10 buckets, S3 Express blocked
  • Reality: function has zero callers. Never enforced. Wiki documented rules that don't exist.

LSP — s3_limits.py

  • Calls find_references(check_s3_bucket_count)
  • Result: zero callers — dead code confirmed
  • Conclusion: not enforced → no wiki update
  • No incorrect content written.
Graph maps:  source file → wiki concept  ·   LSP maps:  source file → execution path → wiki page
